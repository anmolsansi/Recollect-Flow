# TICKET OPE-217: Implement authenticated idempotent capture

## Goal

Provide one reliable endpoint that stores a capture before any downstream work.

## Why This Exists

Capture trust is the product foundation; AI or Notion failure must never lose user input.

## User-Facing Behavior

A new valid capture says “Saved to RecollectFlow.” A retry with the same key says “Already saved.” and returns the original ID.

## Scope

URL/text/note JSON input, bearer auth, strict validation, URL normalization, D1 persistence, idempotent replay, processing-job scheduling, standard responses, and tests.

## Out of Scope

Binary upload, cross-key duplicate merging, extraction, AI, Notion, search, and admin read UI.

## Files to Inspect First

`docs/PRD.md`, `docs/ARCHITECTURE.md`, `migrations/0001_initial.sql`, `apps/worker-api/src/app.ts`.

## Files to Create or Edit

`apps/worker-api/src/captures/*`, `apps/worker-api/src/shared/*`, `apps/worker-api/test/*`, README example.

## Data Model

Insert one `items` row. After success, add one `processing_jobs` row and one audit event. Scheduling failure does not remove the item.

## API Contract

`POST /api/v1/captures`. Required: `idempotency_key`, `source_type`, `source_app`, `captured_at`, `client`. URL source requires `url`; text/note requires nonblank `shared_text`. Optional: `user_reason`, `quick_category`, `privacy_level`. `201` new, `200` replay, `400` malformed JSON, `401` invalid token, `422` invalid fields, `503` storage unavailable. Every response has `meta.request_id`.

## UI States

Shortcut maps new save, replay, fix-input, unauthorized/configuration error, and retry-safe service failure.

## Validation Rules

Reject unknown fields; idempotency key 12–200 characters; URL max 2048; text max 100,000; reason max 2,000; ISO timestamp with offset; controlled enums.

## Error Handling

Use stable codes `INVALID_JSON`, `UNAUTHENTICATED`, `VALIDATION_ERROR`, `STORAGE_UNAVAILABLE`, `INTERNAL_ERROR`. Never log tokens/raw content.

## Auth & Permissions

Capture or admin bearer token may create; comparison is constant-time; tokens come from Worker secrets.

## Dependencies

OPE-216 and approved OPE-213 scope.

## Integration Contract

D1 repository implements find/create/schedule. The service orders find → create → best-effort schedule. Downstream providers are absent from the request path.

## Implementation Steps

Define schema; add auth/error middleware; add repository; implement raw-first service; mount route; add URL normalization; test all states; run full gate.

## Test Cases

New save; same-key replay; invalid token; missing conditional field; malformed JSON; storage failure; scheduler failure after save; tracking URL normalization.

## Manual QA

Start Wrangler against migrated local D1, send README curl twice, confirm `201` then `200` with one item ID.

## Acceptance Criteria

Valid input returns stable ID; invalid input returns structured 4xx; retry creates no second item; scheduler failure still returns saved; tests and migration pass.

## Definition of Done

Code review finds no auth/data-loss/secret issue, `npm run check` passes, local D1 migration passes, docs and Linear match the frozen contract.

## Do Not Change

`/api/v1` prefix, raw-before-derived ordering, separate tokens, D1 authority, and RAG deferral.

## Sequencing Rules

Complete before OPE-218, Shortcut, Notion sync, or enrichment.

## Build Order

1. Contract/schema 2. auth/errors 3. repository 4. service/route 5. tests 6. verification.

## Puzzle-Piece Test

OPE-218 can add conservative cross-key duplicate detection without changing auth, envelopes, or raw persistence semantics.
