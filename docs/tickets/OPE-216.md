# TICKET OPE-216: Add the durable D1 data model

## Goal

Create the source of truth for captures and independent processing/sync history.

## Why This Exists

No enrichment or review surface may become authoritative.

## User-Facing Behavior

Saved items survive downstream failures and can be recovered later.

## Scope

Items, attachments, processing jobs, sync attempts, provider usage, audit events, constraints, indexes, and rollback guidance.

## Out of Scope

FTS5, chunks, embeddings, RAG, production data migration.

## Files to Inspect First

`docs/PRD.md`, `docs/ARCHITECTURE.md`, `wrangler.toml`.

## Files to Create or Edit

`migrations/0001_initial.sql`, `docs/sql/rollback_0001_initial.sql`.

## Data Model

`items` is parent; child foreign keys restrict implicit deletion; `idempotency_key` is unique; state/privacy enums use checks; useful query fields are indexed.

## API Contract

None directly; OPE-217 consumes this schema.

## UI States

None.

## Validation Rules

Required IDs/timestamps, nonnegative counts/sizes, controlled status/privacy values.

## Error Handling

Forward migration must fail loudly; rollback is manual and explicit.

## Auth & Permissions

D1 is accessed only through Worker bindings.

## Dependencies

OPE-215.

## Integration Contract

Repository SQL names match migration columns exactly; rollback SQL must not live under `migrations`.

## Implementation Steps

Define entities; add constraints/indexes; add forward migration; add manual rollback; apply to clean local D1.

## Test Cases

Migration applies; unique idempotency constraint; foreign keys and indexes exist; invalid enum/count constraints reject writes.

## Manual QA

Run `npm run db:migrate:local` on a clean local state and inspect tables.

## Acceptance Criteria

Forward migration executes successfully and represents retries separately from items.

## Definition of Done

Schema, rollback guidance, and migration evidence are committed and documented.

## Do Not Change

D1 authority, raw/derived separation, no embedding tables.

## Sequencing Rules

Complete before OPE-217.

## Build Order

1. Items 2. children 3. indexes 4. rollback 5. local validation.

## Puzzle-Piece Test

OPE-217 can persist a raw item and enqueue a job without schema changes.
