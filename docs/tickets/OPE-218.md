# OPE-218 implementation evidence

- Canonical matching: exact source URL, normalized canonical URL, or exact SHA-256 text hash.
- Storage invariant: one `items` row and one `capture_events` row per accepted new idempotency key.
- Concurrency invariant: unique D1 deduplication-key claims cover source URL, canonical URL and content hash; a claim race is retried as a duplicate capture event. Existing duplicate rows can migrate without destructive consolidation.
- Notes: every event stores its own immutable `user_note`.
- Response: `duplicate_of` returns the reused canonical item ID.
- Boundaries: fragments and the documented tracking allowlist are removed; meaningful query parameters, paths, text case, whitespace, punctuation and encoding remain distinct.
- Automated coverage: standard sites, Instagram, YouTube, exact text, URL-plus-text, different notes, near matches, replay, malformed URL, storage and scheduler failures.
- Live local evidence: two differently keyed normalized URLs returned one canonical capture ID; D1 reported one Item, two CaptureEvents and two distinct notes.
- Production evidence: Worker version `973b4b24-5073-4d4d-aaf3-6b2ce1dd3e1c` accepted two differently keyed normalized URL shares, returned the same canonical item and `duplicate_of`, and D1 confirmed two capture events with two distinct notes.
